# PRO-OPS-001 作業安全程序

| 項目 | 詳情 |
|------|------|
| 編號 | PRO-OPS-001 |
| 版本 | 1.0 |
| 生效日期 | 2026-01-21 |
| 撰寫人 | 謝正強 |
| 審核人 | 資訊安全委員會 |
| 批准人 | 總經理 |
| 適用範圍 | 伺服器、網路設備、作業系統、應用與雲資源的日常運維 |

---

## 1. 目的
建立日常運維、變更、備份、日誌與惡意程式處置的一致流程，降低營運風險。

## 2. 角色與責任（RACI）

| 活動 | 系統/網工 | 服務負責人 | 資安 | 變更審核 | 稽核 |
|------|----------|-----------|------|----------|------|
| 例行維運（補丁、重啟、資源調整） | R | C | C | A | I |
| 變更管理（計畫、審核、回復） | R | C | C | A | I |
| 日誌收集與檢視 | R | I | C | C | I |
| 惡意程式處置 | R | I | A/C | C | I |
| 備份與還原測試 | R | C | C | C | I |

## 3. 程序流程（摘要）
1) 變更計畫：建立變更單，含影響/風險/回復方案，經審核後排程。  
2) 補丁與版本：依風險分級制定更新頻率，關鍵系統需回復點或快照。  
3) 日誌：集中收集與保存，設定告警門檻並定期檢視。  
4) 惡意程式：隔離 → 驗證 → 根因分析 → 補救 → 復原，保留證據。  
5) 備份：依 RPO/RTO 執行備份與還原測試，記錄結果。  
6) 變更關閉：確認服務正常，更新 CMDB/資產清冊。

## 4. 控制要求
- 重大變更需維護時間窗口與回復計畫。  
- 補丁失敗或延期須有風險接受與補救時程。  
- 日誌保存期限：關鍵系統 ≥1 年，其餘 ≥90 天；集中到 SIEM/SOC。  
- 惡意程式樣本與證據需安全保存，避免二次感染。  
- 備份必須加密並定期抽測還原成功率。

## 5. 例外管理
- 無法即時套用補丁時，需評估風險並採取補強（隔離、WAF、MFA）。  
- 例外需設定期限並重新評估。

## 6. 記錄與保存
- 變更管理紀錄（≥3 年）
- 系統日誌與事件日誌（≥1 年）
- 惡意軟體事件紀錄（≥1 年）
- 備份執行與還原測試紀錄（≥1 年）

## Appendix A：變更申請、審核與回復步驟

### 變更分類
| 類型 | 定義 | 審核要求 | 測試要求 |
|------|------|---------|---------|
| 標準變更 | 例行、低風險、已有SOP | IT主管核准 | 非必要 |
| 一般變更 | 中等影響、需規劃 | 變更審核會核准 | 建議 |
| 重大變更 | 高影響、停機、架構調整 | 管理層+變更委員會 | 必須 |
| 緊急變更 | 修復重大事件 | 事後補核 | 視情況 |

### 變更流程
1. 提出：填寫《變更申請單》（FRM-OPS-001）
2. 評估：影響分析、風險評估、回復計畫
3. 核准：依分類取得相應核准
4. 排程：選擇維護時間窗口
5. 執行：依計畫執行並記錄
6. 驗證：確認變更成功且無異常
7. 關閉：更新CMDB/文件，歸檔工單

### FRM-OPS-001 變更申請單

**系統變更申請表**

| 欄位 | 填寫說明 |
|------|---------|
| 變更編號 | 系統產生 |
| 申請日期 | YYYY-MM-DD |
| 申請人 | 姓名、部門、分機 |
| 變更類型 | □ 標準 □ 一般 □ 重大 □ 緊急 |
| 變更描述 | 詳細說明要變更的內容 |
| 變更理由 | 業務需求/問題修復/改進/法規 |
| 影響系統 | 受影響的系統、服務、資料 |
| 影響範圍 | 使用者數量、業務流程、停機時間 |
| 風險評估 | 可能的風險與發生機率 |
| 回復計畫 | 變更失敗時的退回步驟 |
| 測試計畫 | 變更前/後的測試驗證 |
| 執行時間 | 預計開始與結束時間 |
| 執行人員 | 負責執行的技術人員 |
| 主管核准 | IT主管/服務負責人 |
| 變更委員會 | （重大變更）主席簽核 |
| 執行結果 | 成功/失敗/部分完成 |
| 實際完成時間 | |
| 異常說明 | 如有異常或偏差記錄 |

**回復計畫範本**：
1. 觸發條件：何種情況下執行回復
2. 回復步驟：step-by-step指令
3. 回復時間：預估回復所需時間
4. 驗證方式：如何確認已成功回復

## Appendix B：補丁管理與自動化部署流程

### 補丁分類與時限
| 嚴重性 | 定義 | 測試時限 | 部署時限 |
|--------|------|---------|---------|
| Critical | 遠端執行、零日漏洞 | 2天 | 7天內 |
| High | 重大安全漏洞 | 7天 | 30天內 |
| Medium | 一般安全更新 | 14天 | 60天內 |
| Low | 功能改進、非安全 | 視情況 | 90天內或延後 |

### 補丁流程
1. **監控**：訂閱廠商安全通告、使用漏洞掃描工具
2. **評估**：確認適用性與影響範圍
3. **測試**：於測試環境驗證相容性
4. **核准**：依變更管理程序取得核准
5. **部署**：使用WSUS/SCCM/Ansible等工具自動化推送
6. **驗證**：確認部署成功率與系統穩定性
7. **記錄**：更新補丁管理清冊

### 例外處理
- 如無法即時套用，需評估風險並採取補強措施
- 補強措施：網路隔離、WAF規則、存取限制、MFA
- 例外需設定到期日並定期複審

## Appendix C：日誌收集、告警與檢視步驟

### 日誌收集範圍
- **系統日誌**：Windows Event Log、Linux Syslog
- **應用日誌**：Web Server、Database、業務應用
- **安全日誌**：防火牆、IDS/IPS、WAF、EDR
- **存取日誌**：VPN、特權帳號、檔案伺服器

### 日誌集中化
- 使用SIEM/日誌管理平台（如Wazuh、Splunk、ELK）
- 時間同步（NTP）確保時戳一致性
- 加密傳輸與儲存
- 設定適當的保存期限（關鍵系統≥1年，其他≥90天）

### 告警設定
| 事件類型 | 觸發條件 | 通知對象 | 回應時間 |
|---------|---------|---------|---------|
| 登入失敗 | 5次/5分鐘 | IT+資安 | 即時 |
| 特權使用 | 任何使用 | 資安 | 即時 |
| 異常流量 | 超過基準線2倍 | 網管+資安 | 15分鐘 |
| 惡意程式 | 任何偵測 | IT+資安 | 即時 |
| 設定變更 | 防火牆/AD | IT+資安 | 即時 |

### 定期檢視
- **每日**：檢視Critical/High告警
- **每週**：檢視Medium告警與趨勢分析
- **每月**：完整日誌審查與報告

## Appendix D：惡意程式處置（隔離、分析、復原）

### Phase 1：偵測與隔離
1. **偵測來源**：防毒軟體、EDR、使用者通報
2. **立即隔離**：
   - 中斷網路連線（拔網路線或disable網卡）
   - 不要關機（避免記憶體證據消失）
   - 標示為隔離狀態
3. **通報**：依事件管理程序通報資安與主管

### Phase 2：分析與確認
1. **取得樣本**：安全地複製惡意程式樣本
2. **分析**：
   - 使用VirusTotal等線上服務分析
   - 確認變種與家族
   - 評估破壞能力（勒索、後門、竊資）
3. **影響評估**：確認是否有資料外洩或橫向移動

### Phase 3：清除與復原
1. **全面掃描**：使用多款防毒工具
2. **清除殘留**：刪除惡意程式、登錄檔、排程任務
3. **補丁與強化**：更新系統補丁、關閉漏洞
4. **密碼重置**：變更受影響系統的密碼
5. **復原驗證**：確認系統乾淨後才能連回網路
6. **持續監控**：觀察7天確保未再次感染

### Phase 4：事後改進
- 分析感染途徑（釣魚郵件、漏洞、USB）
- 更新防護政策與規則
- 加強員工教育訓練
- 完成事件報告與檢討

## Appendix E：備份排程與還原測試指引

### 備份策略（3-2-1原則）
- **3份副本**：1份正本 + 2份備份
- **2種媒體**：本地備份 + 雲端/異地備份
- **1份離線**：離線備份（防勒索）

### 備份排程範例
| 系統 | 完整備份 | 增量備份 | 保存期限 | RPO | RTO |
|------|---------|---------|---------|-----|-----|
| AD/DNS | 每週日 | 每日 | 30天 | 24hr | 4hr |
| ERP資料庫 | 每日凌晨 | 每6hr | 90天 | 6hr | 8hr |
| 檔案伺服器 | 每週日 | 每日 | 60天 | 24hr | 8hr |
| Email | 每日 | N/A | 365天 | 24hr | 24hr |

### 還原測試
- **頻率**：每季至少測試1次關鍵系統
- **測試項目**：
  - [ ] 備份完整性驗證（檢查碼、掃描錯誤）
  - [ ] 還原至測試環境
  - [ ] 驗證資料正確性與可用性
  - [ ] 記錄還原時間
  - [ ] 記錄遇到的問題與改進
- **記錄**：完成《備份還原測試紀錄》

### 異常處理
- 備份失敗：即時告警並於4小時內解決
- 連續失敗：升級為事件處理
- 儲存空間不足：提前規劃擴充
