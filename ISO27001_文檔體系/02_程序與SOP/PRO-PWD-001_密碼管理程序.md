## 1. 目的
規範密碼設定、變更、重置與保管要求，確保帳號安全。

## 2. 角色與責任（RACI）

| 活動 | 使用者 | 部門主管 | IT/系統管理員 | 資安 | 稽核 |
|------|--------|----------|---------------|------|------|
| 密碼設定/變更 | R | I | C | C | I |
| 密碼重置 | C | I | R | C | I |
| 特權密碼管理 | I | C | R | C | I |
| 密碼政策維護 | I | I | C | A/R | I |

## 3. 程序流程（摘要）
1) 設定：建立帳號後，使用者首次登入即強制改密。  
2) 變更：至少每 90 天變更；高風險系統可縮短為 60 天。  
3) 重置：需身份驗證（工單/主管確認/多因子），臨時密碼 24 小時內必須改密。  
4) 特權密碼：集中保管於密碼庫（需審批與審計），共用密碼需會話記錄。  
5) 停用：帳號停用時，同步失效其憑證/密碼。

## 4. 控制要求
- 長度 ≥ 12，含大小寫、數字、特殊字元，禁止重複近 5 次歷史。  
- 禁用弱密碼（字典、生日、公司名、連續字元）。  
- 禁止共用密碼，禁止明文存放或口頭傳遞。  
- 密碼庫存取需多因子與審計日誌。

## 5. 例外管理
- 如系統限制無法達標，須提出例外申請並在風險可接受內定期補強。  
- 例外需設定到期日並重驗。

## 6. 記錄與保存
- 密碼政策執行紀錄（≥2 年）
- 密碼重置紀錄（≥1 年）
- 密碼庫存取與審計紀錄（≥1 年）

## Appendix A：使用者密碼變更操作指引

### A.1 步驟

**步驟 1：登入前準備**
- 使用 Windows/Linux 內建密碼變更功能或所屬系統自服務入口
- 準備新密碼，確認符合複雜度要求（12+字元、大小寫、數字、特殊字元）
- 進行測試（私人環境驗證）無誤

**步驟 2：執行密碼變更**

Windows/AD 系統
1. 按 Ctrl+Alt+Delete → 選「變更密碼」
2. 輸入舊密碼（當前密碼）
3. 輸入新密碼（符合複雜度）
4. 確認新密碼（重新輸入）
5. 按 Enter 確認，收到「密碼已成功變更」提示

Linux/Unix 系統
1. 開啟終端機
2. 執行 `passwd` 命令
3. 輸入舊密碼驗證身份
4. 輸入新密碼
5. 確認新密碼（重新輸入）
6. 收到「password updated successfully」提示

Web 應用/系統自服務入口
1. 登入應用首頁
2. 進入「帳號設定」或「變更密碼」功能
3. 輸入舊密碼驗證
4. 輸入新密碼並確認
5. 點擊「保存」或「確定」按鈕

**步驟 3：驗證與確認**
- 變更完成後，點擊「登出」
- 使用新密碼重新登入，確認可正常登入
- 若登入失敗，立即通知 IT 部門並申請密碼重置

### A.2 常見問題與解決

| 問題 | 原因 | 解決方案 |
|------|------|---------|
| 密碼不符合複雜度 | 缺少必要字元 | 確保含大小寫、數字、特殊字元 |
| 密碼拒絕更新 | 密碼與歷史相近 | 使用完全不同的密碼 |
| 新舊密碼相同 | 系統禁止 | 設定真正新的密碼 |
| 登入失敗 | 密碼輸入錯誤 | 重新檢查大小寫；如多次失敗，申請重置 |
| 忘記新密碼 | 人為失誤 | 立即聯繫 IT 申請重置 |

### A.3 補充說明
- 密碼變更應在私人、安全的環境中進行（不含肩頸衝浪）
- 不要在公共場所或他人可見的地方變更密碼
- 變更密碼後妥善記錄，勿紀錄於文檔或貼紙
- 定期（至少每 90 天）主動變更密碼，不待系統強制

## Appendix B：密碼重置驗證與發放流程

### B.1 申請流程

**申請人申請**
1. 致電或線上提交 IT Service Desk
2. 提供：帳號名稱、姓名、員工編號（驗證身份）
3. 說明遺忘原因（如遺忘、帳號鎖定、新進員工首次設定）

**IT 部門驗證**
1. 檢查申請人身份（員工編號、部門、報告線）
2. 若可信度不足，致電部門主管確認
3. 驗證無誤後，產生臨時密碼（系統自動或管理員手動）

**臨時密碼發放**
1. **郵件方式**（優先）：寄送至註冊公務郵箱，含臨時密碼與登入說明
2. **電話方式**（若郵件失效）：致電確認身份後告知臨時密碼
3. **親領方式**（特殊情況）：申請人親自至 IT 部門出示身分證件領取

**使用者首次登入**
1. 使用臨時密碼登入系統
2. **系統強制**提示「必須變更密碼」
3. 使用 Appendix A 流程設定新密碼
4. 確認新密碼設定完成

### B.2 安全要求

**臨時密碼特性**
- 長度：≥8 字元
- 複雜度：包含大小寫、數字、特殊字元
- 有效期：24 小時內必須變更，逾期自動失效
- 單次使用：首次登入後即自動失效

**發放記錄**
- 記錄發放時間、發放人員、申請原因、臨時密碼類型
- 保存紀錄 ≥1 年供稽核

### B.3 例外處理

**帳號被鎖定**（連續登入失敗超過 5 次）
- 等待自動解鎖（通常 15 分鐘）或聯繫 IT 手動解鎖
- 解鎖前驗證身份並檢查原因（如被入侵跡象）

**多次重置申請**（同一帳號 1 個月內 > 3 次）
- IT 部門評估是否存在帳號安全風險
- 考慮要求申請人完成身份驗證培訓或啟用多因子認證

## Appendix C：特權密碼庫申請、借出與歸還

### C.1 特權帳號定義

| 帳號類型 | 範例 | 存儲位置 |
|---------|------|--------|
| **OS 根帳號** | root (Linux), Administrator (Windows) | 密碼庫 |
| **資料庫管理** | dba, dbadmin, sa | 密碼庫 |
| **網路設備** | 交換機 admin, 防火牆 admin | 密碼庫 |
| **應用管理** | 應用系統 admin 帳號 | 密碼庫 |
| **雲平台** | AWS IAM root, Azure Global Admin | 密碼庫 |

### C.2 密碼庫系統

**使用工具**
- 企業密碼庫：HashiCorp Vault / CyberArk / Thycotic Secret Server（若已部署）
- 臨時方案：加密檔案存儲於安全位置，定期審計存取

**存儲要求**
- 所有特權密碼加密存儲（至少 AES-256）
- 禁止以明文存放或口頭傳遞
- 定期更換特權密碼（建議 90 天一次）
- 記錄所有存取與異動

### C.3 借出流程

**申請**
1. IT 部門人員或授權管理員提交「特權密碼借出申請單」
2. 說明用途（如伺服器維護、故障排查、系統升級）
3. 指定需要的帳號與系統

**核准**
1. 資訊安全經理或 IT 主管審核申請
2. 確認申請人身份與用途的合理性
3. 核准或拒絕，記錄決策理由

**密碼發放**
1. 授權人員（密碼庫管理員）臨時解鎖密碼
2. 於受限環境下（加密通訊或親領）將密碼發放給申請人
3. 記錄發放時間、發放人、接收人、目的

**使用**
1. 申請人僅在核准用途內使用密碼
2. 禁止將密碼分享給他人
3. 使用期間所有操作自動記錄（session recording）

**歸還**
1. 使用完畢後立即通知密碼庫管理員
2. 提交使用報告（操作摘要、執行結果）
3. 密碼庫管理員確認使用完成並重新加密/變更密碼
4. 記錄歸還時間與確認人簽名

### C.4 審計要求

**定期審計**
- 每月審查密碼庫存取日誌，確認只有授權人員訪問
- 每季確認特權帳號密碼仍符合複雜度要求
- 每年確認密碼庫系統功能正常（加密、審計、告警均工作）

**異常告警**
- 密碼庫登入失敗 > 5 次立即告警
- 單一人員短期內頻繁借出密碼立即審查
- 密碼泄露跡象立即變更並通知相關系統管理員

### C.5 文件與紀錄

**必要文件**
- 特權帳號清冊（含帳號名、系統、所有者、用途）
- 密碼借出申請表（含申請人、核准人、申請時間、歸還時間）

**保存期限**
- 借出申請紀錄：≥1 年
- 密碼庫稽核報告：≥1 年
- 異常事件日誌：≥3 年
