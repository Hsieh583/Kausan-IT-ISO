## 1. 目的
確保在災難或重大中斷時可於可接受時間內恢復關鍵服務，減少營運影響。

## 2. 角色與責任（RACI）

| 活動 | 業務單位 | IT/DR | 資安 | 管理層 | 稽核 |
|------|---------|-------|------|-------|------|
| BIA/風險評估 | R | C | C | A | I |
| BCP/DR 計畫編寫與維護 | C | R | C | A | I |
| 備份與還原測試 | I | R | C | C | I |
| 事件啟動/通報 | R | R | C | A | I |
| 事後檢討 | R | R | C | A | I |

## 3. 程序流程（摘要）
1) BIA：完成關鍵業務 RTO/RPO 需求與資源清單。  
2) 計畫：制定 BCP/DR 計畫，涵蓋通訊、角色、復原步驟與供應商聯繫。  
3) 備份：落實定期備份（含異地/離線），保存與加密。  
4) 演練：每年至少 1 次 DR 演練，含還原測試與通訊演練。  
5) 啟動：依事件等級啟動指揮鏈，記錄時間線與決策。  
6) 事後檢討：完成 RCA、改進計畫與責任分工。

## 4. 控制要求
- 計畫文件須可在災難時離線/紙本取得。  
- 備份加密、完整性驗證與離線/異地保存。  
- 演練結果須形成報告並追蹤缺口改善。  
- 供應商 DR 能力需審核並納入合約要求。

## 5. 例外管理
- 無法滿足 RTO/RPO 時，需管理層書面接受風險與補強措施。  
- 例外需定期複核。

## 6. 記錄與保存
- BIA 報告、BCP/DR 計畫、備份執行與還原測試紀錄、異地備份紀錄、演練報告、事後檢討報告（≥3 年）

## Appendix A：DR 啟動與通報腳本

### A.1 事件通報流程

**第 0 分鐘：事件發現與初步判斷**
1. 系統或員工發現異常（服務中斷、嚴重錯誤、災害警報）
2. 立即檢查：
   - 是否為計畫維護？否→進行 3
   - 影響範圍（單一系統、區域網路、整個資料中心）
   - 預估恢復時間
3. 通報 IT Service Desk（電話）
4. Service Desk 評估事件等級：
   - **P1（致命）**：資料中心火災、地震、完全停電 → 啟動 DR
   - **P2（高）**：部分系統無法 4 小時內恢復 → 通知管理層
   - **P3（中）**：一般故障，預期幾小時內復原 → 常規處理

**第 0-30 分鐘：通報管理層與組織 DR 團隊**

P1 事件（致命）：
1. Service Desk 主管立即（0 分鐘）致電 IT 總監
2. IT 總監致電資安經理與業務單位主管
3. 會議方式：電話會議或線上會議（Zoom/Teams）
4. 會議內容確認：
   - 事件確認與 RTO/RPO 評估
   - 是否啟動 DR？立即決策
5. 決策：
   - **啟動 DR**：轉到 A.1.2 完整啟動流程
   - **持續監控**：保持 15 分鐘回報一次狀態

**啟動確認與指揮中心成立**

啟動指揮中心（應急指揮室或虛擬）
1. 任命指揮官（通常為 IT 總監或業務副總）
2. 召集 DR 團隊成員（參考 PRO-BCP-001 主文中的組織）
3. 建立通訊頻道（電話會議、Slack 頻道、群組郵件）
4. 分配責任（見 A.1.3 角色表）
5. 確認供應商聯繫清單已激活

### A.2 通報清單與話術

**對內通報（IT 團隊與內部管理層）**

| 對象 | 通報時間 | 通報方式 | 內容 | 頻率 |
|------|--------|--------|------|------|
| IT 主管 | 即時 | 電話 | 事件簡述、初步影響、建議行動 | 發生時 |
| 指揮官 | 即時 | 電話 | 確認啟動 DR、集結人員、進入應急模式 | 1 次 |
| DR 團隊 | 即時 | 簡訊+電話+會議 | 事件概述、你的任務、集結地點/虛擬會議連結 | 1 次 |
| 管理層（董事長、總經理） | 0-30 分鐘 | 電話 | 事件確認、預估影響、恢復時間表、需求（如公開聲明） | 每 4 小時 |

**對外通報（客戶、供應商、媒體）**

| 對象 | 通報時間 | 通報方式 | 內容 | 頻率 |
|------|--------|--------|------|------|
| 主要客戶 | 事件後 4 小時 | 郵件+電話 | 服務中斷確認、預估恢復時間、聯繫人 | 首次通知 + 每 4 小時更新 |
| 供應商 | 事件後 2 小時 | 電話 | 尋求支援的明確請求（如資料中心、頻寬等） | 1 次（若需要） |
| 媒體/公眾 | 事件後 6 小時（如涉及個資洩露） | 官方聲明 | 由發言人統一對外發言 | 視情況 |

**通報話術範本**

*對 IT 團隊（第 0 分鐘）：*
> "資料中心偵測到火警，請立即停止所有操作。我們啟動 DR 計畫。所有系統管理員 10 分鐘內進入 Zoom 會議 [連結]。帶上筆記本與備用電源。"

*對主要客戶（第 4 小時）：*
> "我們的系統目前因 [簡述] 無法提供服務。我們已啟動災害復原計畫，預計於 [時間] 恢復。期間如有緊急需求，請撥打 [緊急聯繫電話]。感謝您的耐心。"

*對管理層（每 4 小時一次）：*
> "狀態更新：[恢復進度 %]。預估還需 [X] 小時完成。已通知 [客戶數] 個主要客戶。目前需求：[資源/支援]。下次報告：[時間]。"

### A.3 聯絡清單（激活檢查）

**IT 團隊聯絡清單**
- [ ] IT 總監手機
- [ ] 資安經理手機
- [ ] 備用系統主管手機
- [ ] 資料庫管理員 (2 人以上)
- [ ] 網路管理員
- [ ] 虛擬環境管理員
- [ ] 備份管理員
- **驗證**：每季更新一次，確保所有電話號碼有效

**客戶聯絡清單**
- 主要客戶前 5 名（帳戶經理、技術聯絡人電話/郵件）
- 中等客戶分組聯絡人（部門主管郵件清單）
- **驗證**：每季更新，刪除已離職客戶

**供應商與外部支援**
- 資料中心廠商 24 小時緊急電話
- ISP 緊急號碼
- 雲平台供應商（AWS/Azure）緊急支援
- **驗證**：確保已簽訂 SLA 或支援合約

## Appendix B：系統與資料還原步驟

### B.1 整體還原流程圖

```
事件確認 (0-30分鐘) 
      ↓
啟動 DR (30分鐘)
      ↓
[平行執行]
├─ IT: 驗證備份完整性
├─ 業務: 通知客戶
└─ 資安: 檢查備份無感染
      ↓
還原虛擬環境與資料庫 (1-4小時)
      ↓
還原應用系統與配置 (2-6小時)
      ↓
資料驗證與功能測試 (1-2小時)
      ↓
復原通知與切換 (0.5-1小時)
      ↓
事後監控與檢驗 (4-24小時)
```

### B.2 優先級順序與 RTO 目標

| 優先級 | 系統 | RTO 目標 | RPO 目標 | 還原資源 | 備註 |
|-------|------|--------|---------|--------|------|
| **P1-物流** | 訂單系統、出貨排程、車隊調度 | 4 小時 | 1 小時 | 虛擬環境、備份資料庫、應用程式 | 立即啟動 |
| **P1-客戶** | 客戶管理系統、發票系統 | 8 小時 | 2 小時 | 虛擬環境、備份資料庫 | 1 小時後啟動 |
| **P2-財務** | 會計系統、應付應收 | 24 小時 | 4 小時 | 虛擬環境、備份資料庫 | 4 小時後啟動 |
| **P3-辦公** | 郵件、檔案伺服器、內部工具 | 48 小時 | 24 小時 | 虛擬環境、雲端同步 | 次日啟動 |

### B.3 資料庫還原步驟

**準備（15 分鐘）**
1. 確認災害備份位置與檔案完整性（查驗 MD5）
2. 驗證最近 1 小時備份存在且無損毀
3. 確認還原目標伺服器硬體資源充足（磁碟、記憶體）

**還原執行（30-60 分鐘，取決於資料量）**
1. 啟動資料庫伺服器（物理機或虛擬機）
2. 掛載備份媒介或連線至備份存儲
3. 執行還原命令：
   ```bash
   # SQL Server 範例
   RESTORE DATABASE [DatabaseName] FROM DISK = 'Path\backup.bak'
   
   # MySQL 範例
   mysql -u root -p < backup_full.sql
   
   # PostgreSQL 範例
   pg_restore -d databasename backup.dump
   ```
4. 監控還原進度（通常 10-60 分鐘，視資料大小）
5. 驗證還原完成，檢查完整性與一致性

**驗證（15-30 分鐘）**
1. 執行資料庫完整性檢查（DBCC CheckDB / MySQL CHECK / PostgreSQL pgrepair）
2. 測試關鍵表、索引、預存程序可用性
3. 驗證資料時間戳記是否符合 RPO（應在 1 小時內）

### B.4 應用系統還原步驟

**前置檢查（10 分鐘）**
1. 虛擬環境已啟動並分配資源
2. 網路連線、DNS 解析正常
3. 必要的系統軟體已安裝（.NET Framework、Java、Web Server）

**還原執行（30-90 分鐘）**
1. 從備份還原應用程式檔案至應用伺服器
2. 驗證應用程式版本與上次備份一致
3. 檢查配置檔（web.config、app.config）指向備份的資料庫
4. 啟動應用程式服務
5. 確認應用日誌無異常錯誤

**功能測試（30-60 分鐘）**
1. 測試應用首頁能否載入
2. 執行關鍵業務流程測試（如建立訂單、查詢客戶）
3. 測試報表與匯出功能
4. 驗證外部介面正常（如第三方 API、郵件發送）
5. 記錄測試結果，簽核通過後進行切換

### B.5 網路與 DNS 切換（5-15 分鐘）

**DNS 更新**
1. 更新 DNS A 記錄，指向備援伺服器 IP
   ```bash
   # 範例
   example.com A 192.168.2.50  (備援伺服器)
   ```
2. 縮短 TTL 至 5 分鐘加快傳播（若事前未設定，此時無法立即生效）
3. 確認 DNS 全球生效（可用 nslookup 或 dig 驗證）
4. 監控客戶端是否成功解析至新 IP

**通知與驗證**
1. 發送「服務復原」通知至客戶（包含驗證連結）
2. 抽樣測試客戶能否正常連線並使用系統
3. 監控應用日誌，確認無異常錯誤

## Appendix C：備份完整性驗證與異地搬移作業

### C.1 備份完整性驗證

**日常驗證（每日）**
1. 檢查備份任務執行狀態
   - Veeam 控制台查看昨晚備份是否成功 ✓
   - 備份檔案大小是否在預期範圍（偏離 >20% 需調查）
   - 備份耗時是否異常（超過 SLA 時間）
2. 檢查備份日誌是否含警告或錯誤
3. Wazuh 告警確認（若已配置備份監控告警）

**每月完整還原測試**
1. 選擇非生產環境（如測試環境）進行還原
2. 隨機選擇 1-2 個備份版本（不一定最新）
3. 還原整個虛擬機或資料庫至測試環境
4. 驗證還原資料完整、可用、無損毀
5. 記錄測試結果、耗時、發現的問題
6. 若有問題立即通知備份管理員調查

**每季備份驗證報告**
1. 統計過去 3 個月備份成功率（目標 ≥99%）
2. 統計平均備份大小、耗時、還原耗時
3. 統計失敗備份次數與原因分析
4. 提出改善建議（如增加備份帶寬、調整排程）
5. 生成報告提交管理層

### C.2 備份檔案完整性檢查

**雜湊值驗證**
1. 備份完成後，產生 MD5/SHA-256 雜湊值
   ```bash
   # Windows
   CertUtil -hashfile backup.vib SHA256
   
   # Linux
   sha256sum backup.vib
   ```
2. 紀錄雜湊值於備份日誌
3. 異地傳送前重新計算，確認一致（確保傳輸無損毀）

**檔案完整性驗證工具**
1. Veeam 內建的 Backup File Integrity Check
2. 執行定期掃描（如每月 1 次）
3. 檢測檔案損毀跡象（corrupt sectors, bad blocks）
4. 若發現損毀，立即通知並啟動應急備份

### C.3 異地備份搬移流程

**異地備份策略（3-2-1 原則）**
- 3 份備份：原始資料、本地備份、異地備份
- 2 種媒介：磁帶 + 虛擬機快照 或 磁帶 + 雲端
- 1 份離線：至少 1 份完全離線存儲

**每週異地備份搬移**

時間安排
- 搬移日期：每週五晚間 6 點
- 搬移間隔：7 天（確保最多 7 天資料遺失）
- 搬移方式：自動化或手動（根據工具能力）

搬移步驟
1. **準備備份檔案** (30 分鐘)
   - 確認本週所有備份完成且無誤
   - 計算本週備份總大小
   - 驗證異地存儲空間充足

2. **傳送備份至異地** (1-4 小時，取決於頻寬)
   - 方式 A（自動）：Veeam 或備份軟體自動上傳至遠端站點或雲端
   - 方式 B（手動）：透過加密 VPN/SFTP 上傳
   - 監控傳輸進度與速度
   - 使用進度條或日誌確認所有檔案已到達

3. **異地驗證** (30 分鐘)
   - 遠端站點人員確認檔案已接收
   - 驗證檔案完整性（大小、雜湊值相符）
   - 檢查磁碟空間是否充足
   - 回報確認訊息至本地

4. **紀錄與通知** (15 分鐘)
   - 記錄搬移日期、備份檔案數、總大小、耗時
   - 記錄傳輸狀態（成功/失敗）、失敗原因
   - 發送搬移完成確認郵件至資訊部主管
   - 若有異常立即通知備份管理員

**異地備份日誌範本**

| 日期 | 備份版本 | 檔案數 | 大小 (GB) | 傳輸耗時 | 驗證結果 | 簽署人 | 備註 |
|------|---------|-------|----------|---------|--------|-------|------|
| 2026-01-17 | Weekly-DB-20260117 | 5 | 250 | 2.5 小時 | ✓ 成功 | 李 IT | 雲端上傳 |
| 2026-01-24 | Weekly-DB-20260124 | 5 | 248 | 2.3 小時 | ✓ 成功 | 王 IT | 雲端上傳 |

### C.4 異地備份存儲位置與保持

**物理異地存儲（磁帶）**
- 位置：距離公司 ≥100 公里的安全金庫（銀行保管箱或專業存檔服務）
- 保存溫度：15-25°C，濕度 30-70%
- 每季檢查一次（檢查磁帶標籤完整、無物理損毀）
- 定期輪轉（新磁帶進、舊磁帶出，保持最近 6 個月的備份）

**雲端異地存儲**
- 平台：AWS S3、Azure Blob Storage 或公司指定雲端
- 區域：選擇不同的地理區域（如主站台在台北，備援選日本或新加坡）
- 加密：傳輸中 TLS、儲存中 AES-256
- 成本控制：設置週期刪除政策（如 1 年後自動刪除舊版本）

**訪問控制與審計**
- 僅授權的備份管理員可存取異地備份
- 所有存取操作記錄於日誌（AWS CloudTrail、Azure Activity Log）
- 定期審查存取日誌，異常立即調查

## Appendix D：演練計畫與檢核表

### D.1 年度 DR 演練計畫

**演練目標**
- 驗證 BCP/DR 計畫有效性
- 培訓 DR 團隊技能與協調能力
- 識別計畫中的缺陷與改善點
- 客戶與供應商建立信心

**演練類型與頻率**

| 演練類型 | 頻率 | 時間 | 參與者 | 內容 |
|---------|------|------|-------|------|
| **桌上演練** (Tabletop) | 每季 1 次 | 2 小時 | IT/業務/管理層 | 討論災害情景，驗證計畫的邏輯與決策流程 |
| **還原測試演練** (Restore Test) | 每季 1 次 | 2-4 小時 | IT 技術人員 | 實際進行備份還原，不涉及客戶或生產系統 |
| **全面演練** (Full Scale) | 每年 1 次 | 4-8 小時 | 所有相關單位 | 實際切換至備援環境，驗證完整復原流程 |

**演練時間安排（建議）**

### D.3 演練劇本範例：火災情景

**開始時間**：上午 9:00  
**演練持續時間**：4 小時  
**參與者**：IT 主管、系統管理員 (2 人)、資料庫管理員、網路管理員、業務代表、資安經理

**劇本流程**

**9:00 - 事件通報**
```
主持人宣佈：
"現在時間是 2026 年 6 月 15 日上午 9:00。
我們的台北資料中心偵測到火警警報。
消防隊已獲通報，建物正在疏散。
IT 值班人員已無法進入機房。
我們預計 1 小時內確認建物安全狀況。

各位現在假設資料中心在未來 2 小時內無法恢復使用。
我們需要啟動 DR 計畫，切換至備援環境。

IT 主管：立即確認備份完整性。
資安經理：檢查異地備份無感染。
業務代表：準備客戶通知。
系統管理員：3 分鐘內進入虛擬會議，領取你們的任務。"
```

**9:03 - 任務分配**
| 角色 | 任務 | 目標完成時間 |
|------|------|-----------|
| IT 主管 | 1. 確認最新備份 (30分鐘)  2. 決策啟動 DR (5分鐘)  3. 指揮還原執行 | 9:35 決策 |
| DBA | 還原資料庫至備援環境 | 10:45 |
| 系統管理員 | 還原虛擬機與應用程式 | 11:00 |
| 網路管理員 | 更新 DNS、測試網路連線 | 10:50 |
| 業務代表 | 撰寫客戶通知、電話通知前 3 大客戶 | 9:30 與 11:30 |

**9:30 - 狀態檢查**
```
IT 主管檢查進度：
- 備份確認：✓ 完成（08:00 PM 備份成功）
- 決策：啟動 DR（宣佈於全體會議）
- DBA 任務進度：資料庫還原中，預計 10:30 完成
```

**10:00 - 中途評估與問題處理**
```
網路管理員回報：
"DNS 更新已完成，但某些客戶端 DNS 仍指向舊地址。
TTL 設為 5 分鐘，預計 5-10 分鐘內全球生效。"

觀察員記錄：
- 發現：DNS TTL 預先設置的重要性（應在平時設為 5 分鐘）
- 改善點：日常應將關鍵應用的 DNS TTL 預設為低值
```

**11:00 - 系統驗證**
```
系統管理員測試：
- 應用首頁能否載入？✓ YES（11:15）
- 能否建立新訂單？✓ YES（11:25）
- 舊訂單資料是否完整？✓ YES（11:30，確認資料時間戳為 08:00 PM）
```

**11:30 - 客戶通知與復原確認**
```
業務代表執行客戶通知：
- 主要客戶 A、B、C 已電話確認
- 客戶回饋：資料完整、可正常操作
- 簽署客戶確認單

觀察員評估：
- 優點：客戶通知迅速，語氣專業
- 改善：應事先準備更詳細的故障排除指南提供給客戶
```

**12:00 - 演練結束與紀錄**
```
IT 主管宣佈演練結束。
所有參與者停止操作，保存測試結果與日誌。
觀察員開始匯總評估。
```
