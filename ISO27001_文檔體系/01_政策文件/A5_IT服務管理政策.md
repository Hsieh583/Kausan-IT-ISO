# A5 IT 服務管理政策

**政策編號**: A5  
**政策名稱**: 資訊科技服務管理政策  
**版本**: 1.0  
**生效日期**: 2026年1月24日  
**責任部門**: 資訊技術部  
**上層政策依據**: ISO 9001 (品質管理系統) + ISO 27001 (資訊安全管理系統)

---

## 1. 政策目的

本政策定義資訊技術（IT）部門在組織中的角色、責任、服務邊界，以及服務品質管理的原則。

IT 不是純技術支援部門，而是為組織提供**「穩定、安全、高效的數位服務」**的內部服務單位。IT 的品質管理跨越 ISO 9001（營運品質）和 ISO 27001（資訊安全），形成統一的「服務管理體系」。

---

## 2. 政策範圍

本政策適用於：
- IT 部門的所有日常運作
- IT 投資決策與資源配置
- IT 與其他部門（業務、財會、HR等）的互動
- IT 外包與採購決策
- IT 績效評估與改進

---

## 3. IT 部門的核心角色

### 3.1 使命宣言

> **IT 是組織的數位基礎設施運營商。我們的使命是透過提供可靠、安全、高效的數位服務，支撐組織的營運目標和策略發展。**

### 3.2 核心職能（不是技術清單，而是服務清單）

IT 提供以下 7 項核心服務，而不是零散的技術支援：

| # | 服務名稱 | 價值主張 | 主要客戶 |
|---|--------|--------|--------|
| 1 | **Identity & Access Service** | 讓對的人在對的時間存取對的系統 | HR、各部門、全員 |
| 2 | **Compute & Application Service** | 穩定可用的系統執行環境 | 營運、財會、各系統用戶 |
| 3 | **Data Protection & Recovery Service** | 資料不會不見，出事也救得回來 | 全公司、管理層 |
| 4 | **Security Monitoring & Incident Response Service** | 出事有訊號，能快速處理 | 管理層、IT自己 |
| 5 | **Network & Connectivity Service** | 人跟系統之間不被網路卡住 | 全員、分點、倉庫 |
| 6 | **Productivity & Automation Service** | 讓人少做重複事，工作跑得更快 | 行政、營運、財會、業務 |
| 7 | **IT Governance & Visibility Service** | 讓管理層知道：IT 是被管控的 | 管理層、稽核單位、IT自己 |

**詳細服務定義參考**: [IT Service Catalog（服務語言版）](../../../資訊營運/IT%20Service%20Catalog_服務語言版.md)

### 3.3 IT 的角色邊界

**IT 的責任**：
- ✅ 確保 7 項核心服務的穩定性、安全性、可用性
- ✅ 制定服務標準（SLO）、監控服務品質、改進服務
- ✅ 評估與採購適合的技術/工具/服務
- ✅ 管理 IT 相關的風險與變更
- ✅ 與業務部門溝通、理解業務需求

**IT 的非責任**（避免職能混淆）**：
- ❌ 不負責業務邏輯或業務系統的設計（業務部門責任）
- ❌ 不負責業務流程優化（應由業務部門主導，IT 提供自動化支援）
- ❌ 不負責使用者的個人電腦或軟體盜版管理（由使用者或部門負責）
- ❌ 不負責業務資料的正確性或準確性（資料所有者責任）

## 4. 服務品質管理框架（L1-L5）

IT 服務品質分為 5 個成熟度層級，從「基礎穩定」逐層提升至「卓越治理」。

### 4.1 五層架構

```
L5: 文件與合規證書    ◄── 驗收層（最後才達成）
L4: 工具與自動化      ◄── 效率層（加速交付）
L3: 使用者體驗        ◄── 便利層（人用起來順）
L2: 防禦與監控        ◄── 保護層（出事有訊號）
L1: 可靠性與容灾      ◄── 基礎層（不中斷、不爆炸）
```

**核心原則**：
- **L1 是底線**。如果可靠性不穩定，其他層級都會崩塌
- **L2 是防線**。防禦與監控是 IT 的保險；無法快速偵測與應對的系統等於沒有防禦
- **L3 是黏著劑**。人不用，所有投資都白費；流程太複雜，人就會自創工具
- **L4 是倍增器**。自動化才能規模化，人力有限
- **L5 是驗收**。文件與合規是「驗證」系統運作良好，而非「指導」系統設計

**詳細框架參考**: [IT營運優先層級框架（L1-L5）](../../../資訊營運/L1-L5_營運優先層級框架.md)

### 4.2 成熟度目標（與 ISO 9001 對應）

| 層級 | IT Service Catalog 中的角色 | 對應 ISO 9001 要求 |
|------|---------------------------|------------------|
| **L1** | 可靠性基礎 | 8.1 營運計劃與控制 |
| **L2** | 防禦與監控 | 8.4 外部提供的過程控制；8.5 生產與服務提供控制 |
| **L3** | 使用者滿意度 | 8.2 與顧客相關的要求確定；8.6 產品和服務的控制 |
| **L4** | 過程效率與持續改進 | 10.3 持續改進 |
| **L5** | 品質管理與稽核就緒 | 9.1 監測、測量、分析和評估；9.2 內部審核 |

## 5. Make vs Buy 決策原則（採購政策）

IT 在每項服務上面臨採購決策。原則是：**優先採購成熟服務，精力投入核心差異化領域**。

### 5.1 決策框架

| 決策類型 | 定義 | 適用情境 | 例子 |
|--------|------|--------|------|
| **Buy (採購)** | 採購現成的雲服務或 SaaS | 成熟度高、競爭激烈、無差異化、快速上線 | Azure、AWS、Microsoft 365、Okta |
| **Manage (外包)** | 將運維外包給 MSP 或服務商 | 團隊能力不足、成本效益、24/7 支援需求 | MSP 運維、SOC 監控外包 |
| **Build (自建)** | 自己開發或維護系統 | 核心競爭力、高度定制、現有系統深度整合 | 內部工作流自動化、特定業務系統 |
| **Hybrid (混合)** | 混合以上模式 | 最常見；公雲基礎設施 + 自建應用邏輯 | Azure + 自建應用、AWS + 內部工具 |

### 5.2 採購策略指引

**L1-L3 服務（穩定度要求高）**：
- 優先採購或外包
- 理由：成熟、風險低、專業廠商的成熟度高於內部
- 例：備份、SIEM、身分管理都應優先用大廠成熟方案

**L4 服務（流程優化）**：
- 可混合模式
- 核心邏輯自建（符合組織特性）+ 基礎工具採購
- 例：工作流自動化用 ServiceNow/Power Automate 的平台 + 自定義業務邏輯

**L5 服務（治理與文檔）**：
- 基礎設施採購 + 內容自創
- 例：文件管理系統可以買（Sharepoint、Confluence），但文檔內容要自己寫

### 5.3 評估準則

每項採購決策應評估：

| 評估項 | 權重 | 評分 |
|-------|------|------|
| **成熟度** | 40% | 市場上有成熟方案嗎？ |
| **成本效益** | 30% | 自建 vs 採購的成本對比 |
| **差異化價值** | 20% | 這是我們的核心競爭力嗎？ |
| **風險** | 10% | 採購/外包的風險可控嗎？ |

## 6. 績效指標體系（KPI）

IT 的績效不只看「有沒有安全違規」（27001），也要看「服務品質有沒有達標」（9001）。

### 6.1 按服務分類的 KPI

**Identity & Access Service**:
- 帳號申請完成時間
- 孤兒帳號數量
- 異常登入告警響應時間

**Compute & Application Service**:
- 關鍵系統可用率
- 平均修復時間（MTTR）
- 高風險單點數量

**Data Protection & Recovery Service**:
- 備份成功率
- 還原演練成功率
- 未測試系統比例

**Security Monitoring & Incident Response Service**:
- 高風險事件偵測時間（MTTD）
- 事件平均修復時間（MTTR）
- 高風險事件閉環率

**Network & Connectivity Service**:
- 關鍵連線可用性
- 斷線檢測時間
- 重複問題比例

**Productivity & Automation Service**:
- 自動化覆蓋率
- 人力投入下降比例
- 使用者滿意度

**IT Governance & Visibility Service**:
- 無主責事件數
- 稽核發現缺陷率
- 稽核準備時間

### 6.2 整體 IT 績效評分（月度）

每月評估 IT 的整體成熟度，並設定改進目標：

```
目標成熟度（18 個月後）:
  L1 (可靠性): 3.5/5 ✓
  L2 (防禦): 3.0/5 ✓
  L3 (體驗): 3.0/5 ✓
  L4 (自動化): 3.0/5 ✓
  L5 (治理): 3.0/5 ✓
  ─────────────────────
  平均成熟度: 3.1/5 (「可靠」等級)
```

## 7. IT 與業務的合作原則

### 7.1 需求評估與優先級

業務部門提出 IT 需求時，IT 應評估：

1. **對組織目標的貢獻度** —— 是否支撐策略？
2. **對 L1-L5 的影響** —— 是否改進服務品質？
3. **資源與成本** —— 投資回報率？
4. **風險** —— 會不會破壞現有穩定性？

**優先級排序**（從高到低）：
- 🔴 **P0**: L1 故障或缺陷（系統停機、安全漏洞）
- 🟠 **P1**: L2-L3 改進（防禦強化、使用者體驗優化）
- 🟡 **P2**: L4 效率提升（自動化、流程優化）
- 🟢 **P3**: L5 文檔與合規準備（稽核支援）

### 7.2 SLA 與 SLO

IT 應與業務部門協商清楚服務水準（SLA）：

**例**：
- 關鍵系統可用率 ≥99.5%
- 一般系統可用率 ≥99%
- 帳號申請完成時間 ≤2 天
- 安全告警響應 ≤1 小時

**SLA 未達時的處理**：
- 根本原因分析（RCA）
- 改進計畫（CAP）
- 預防措施（不能單純賠償，要改進）

## 8. 風險管理

### 8.1 IT 風險的三大類別

**L1 風險** —— 可靠性風險
- 系統中斷風險
- 資料遺失風險
- 災難復原失敗風險

**L2 風險** —— 安全風險
- 未偵測的威脅
- 事件響應遲緩
- 審計軌跡不完整

**L3-L5 風險** —— 運營風險
- 使用者不配合（繞過流程）
- 自動化故障
- 稽核不準備

### 8.2 風險應對原則

- **L1 風險**: 重點投資，必須降低 → Buy (採購可靠方案) / HA+DR
- **L2 風險**: 無法規避，需要監控與快速應對 → Buy (成熟 SIEM) / Manage (SOC 外包)
- **L3-L5 風險**: 可部分自建，但不能影響 L1-L2 → Build (內部流程優化)

## 9. 持續改進

### 9.1 改進周期

- **日常**: 事件處理與小改進（緊急修復）
- **週**: 看板更新、優先級調整
- **月**: IT 服務評估、KPI 檢討、L1-L5 成熟度評分
- **季**: 策略評估、採購計劃、外部稽核或供應商評估
- **年**: 政策更新、長期投資規劃、ISO 認證更新

### 9.2 改進來源

改進建議可來自：
- 內部事件後的根本原因分析（RCA）
- 業務部門的反饋
- 使用者滿意度調查
- 稽核發現（內部/外部）
- 業界最佳實踐或新技術

### 9.3 改進追蹤

每項改進都應記錄：
- 改進內容
- 預期效果
- 完成期限
- 責任人
- 完成驗證

## 10. 責任與權限

| 角色 | 職責 |
|------|------|
| **IT Director/Head** | 整體 IT 策略、預算、優先級決策、與高管溝通 |
| **L1-L5 Layer Owner** | 各層的具體執行與成熟度管理 |
| **Service Owner** | 每項服務的 SLA、KPI、改進計劃 |
| **IT Team** | 日常運維與改進執行 |
| **IT Governance Committee** | 月度評估、跨部門協調、風險決策 |

---

## 11. 政策審查與更新

本政策應每年審查一次，或在以下情況下進行修正：

- 組織策略變更
- 新的 ISO 標準或法規要求
- 重大 IT 事件或失敗
- 外部環境變化（如新技術出現）
- 實施經驗反饋
