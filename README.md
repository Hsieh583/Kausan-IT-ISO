# ISO 27001 文件與記錄體系（私有專案）

> 本專案為組織內部 **ISO/IEC 27001:2022 資訊安全管理系統（ISMS）** 之文件與記錄治理專案，僅供授權人員使用。

---

## 一、專案目的

本 GitHub 私有專案作為 **ISO 27001 文件的權威來源（Authoritative Source）**，用於：

* 建立與維護 ISO 27001 所需之政策（Policy）、程序（Procedure）與治理文件
* 以「**最小可行文件集合（Minimum Viable ISMS）**」為原則，避免文件爆量
* 提供清楚、可追溯、可稽核的版本控制機制
* 作為後續 PDF 正式發布至 SharePoint 之前的唯一原始來源

本專案 **不作為記錄（Records）最終保存位置**，僅保存文件本身與必要之範本。

---

## 二、治理原則（必讀）

### 1. 文件類型定義

本專案僅允許以下四種類型文件：

| 類型                | 說明                            |
| ----------------- | ----------------------------- |
| **Policy（政策）**    | 管理承諾與原則，不含操作細節                |
| **Procedure（程序）** | 跨角色流程與責任定義                    |
| **SOP（作業指引）**     | 實際操作方式，**僅能以 Procedure 附錄存在** |
| **Record（記錄）**    | 已發生事實之證據（僅定義，不保存實體）           |

❗ **禁止新增其他文件類型**（如 Guideline、Manual、Note 等）。

---

### 2. SOP 附錄化（強制規則）

* SOP **不得獨立成檔**
* 所有操作步驟一律寫在對應 Procedure 的 `Appendix` 中
* SOP 附錄變更 **不視為主文件改版**

目的在於：

* 控制文件數量
* 降低核准與改版成本
* 提高長期可維護性

---

### 3. 最小文件集合原則

本專案遵循以下原則：

* 能合併者不拆分
* 能用 Procedure 規範者，不另立 Policy
* 記錄優先由系統自動產生（如 AD、Wazuh、Veeam）
* 本專案 **不保存大量操作紀錄或日誌**

## 四、版本控制與發布原則

### 1. 權威來源

* **GitHub（本專案）**：唯一權威來源
* **SharePoint**：僅存放已核准之 PDF 發布版本（唯讀）

任何文件內容差異，**以 GitHub 為準**。

---

### 2. 版本規則（建議）

* 主版本（Major）：治理或控制原則重大變更
* 次版本（Minor）：流程或結構調整
* 修訂版本（Patch）：文字修正、不影響控制

範例：`v1.2.1`

---

## 五、適用對象與使用方式

### 適用對象

* 資訊安全負責人
* IT 管理人員
* 合規 / 稽核協助人員
* 受授權之外部顧問（唯讀）

### 不適用對象

* 一般員工日常操作
* 作業紀錄填寫

---

## 六、注意事項（請務必遵守）

* ❌ 不要在本 Repo 儲存實際資安事件紀錄、帳號清冊、日誌
* ❌ 不要直接修改 SharePoint 上的正式文件
* ❌ 不要為了對應控制條文而「硬拆文件」

若對文件分類或是否需要新增文件有疑問，**以治理文件（P0）為最高判斷依據**。
