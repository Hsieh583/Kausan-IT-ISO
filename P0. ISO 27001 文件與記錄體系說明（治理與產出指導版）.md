# ISO 27001 文件與記錄體系說明（治理與產出指導版）

## 0. 本文件目的與使用方式

本文件為本組織 ISO 27001:2022 資訊安全管理系統（ISMS）之「文件治理母文件」，目的在於：

- 定義文件與記錄之**最小必要集合（Minimum Viable ISMS Documentation）**
- 規範 Policy / Procedure / SOP / Record 之角色與邊界
- 作為後續所有 ISO 文件撰寫、合併、版控與保存之唯一依據

本文件本身亦納入文件管制與定期審查。

---

## 1. 文件類型定義與治理原則

### 1.1 文件類型定義

| 類型 | 定義 | 內容性質 | 核准層級 | 變更頻率 |
|---|---|---|---|---|
| Policy（政策） | 管理承諾與原則 | 不含操作細節 | 最高管理階層 | 低 |
| Procedure（程序） | 跨角色流程與責任 | 含流程、不含工具細節 | 資安委員會 | 中 |
| SOP（作業指引） | 實際操作方式 | 工具與步驟導向 | 資訊部 | 高 |
| Record（記錄） | 已發生事實之證據 | 不可任意修改 | 依權責 | 依保存期限 |

---

### 1.2 SOP 數量控制與附錄化原則（強制）

為避免文件數量失控，SOP 不得獨立成為主要管制文件，原則如下：

- 每一控制領域僅允許 **1 份主 Procedure 文件**
- 所有實際操作說明以 **附錄（Appendix）形式**存在
- 附錄不單獨編號、不需單獨核准
- 附錄變更不視為主文件改版

範例：
- PRO-OPS-001 作業安全程序
  - Appendix A：備份作業指引
  - Appendix B：日誌檢視指引
  - Appendix C：惡意程式處置流程

---

### 1.3 文件最小集合（Minimum Viable ISMS）原則

本組織採取「可稽核、可維運、不可爆量」原則：

- 能合併者不拆分
- 能以程序涵蓋者不另立政策
- SOP 僅作為附錄
- 記錄來源以系統自動產生為優先

---

## 2. 文件權威來源與平台分工

### 2.1 權威來源定義

- **Git 版本庫**：所有政策、程序、標準文件之唯一權威來源（Authoritative Source）
- **SharePoint**：僅存放經核准之正式發佈版本（PDF），禁止直接修改

### 2.2 文件流轉規則

1. 文件於 GitHub 以 Markdown 撰寫與版控
2. 經核准後轉為 PDF
3. PDF 發佈至 SharePoint 作為對內與對稽核版本

## 3. 工具與平台中立性聲明

本文件所提及之技術工具（如 Wazuh、Veeam、AD、VPN 等）為現行實務實例，用以說明控制措施之落地方式。

控制要求本身不依附於任何特定品牌、產品或廠商，未來如工具更換，不視為控制失效，僅需更新實作說明或附錄。

---

## 4. 記錄保存年限訂定原則

各類記錄保存年限之訂定，係依據以下因素綜合評估：

- 法規與合約要求
- 風險等級與稽核需求
- 實務可行性與儲存成本

保存年限於年度管理審查中定期檢討，不視為固定不變。

## 9. 文件使用與後續產出指引（給文件撰寫者）

- 撰寫任何 ISO 文件前，須先確認其屬於 Policy / Procedure / SOP / Record
- 不得新增文件類型
- SOP 一律以附錄方式存在
- 文件數量爭議時，以「能否合併」為優先考量

本文件為後續所有 ISO 27001 文件產出的最高指導原則。
