# 資訊營運（Operational-First IT）

**核心理念**: 不中斷、不爆炸、出事有訊號能處理、人用起來順、最後才是文件與證書

---

## 📋 目錄定位

本目錄是「**營運優先型IT**」的實施指南與管理體系。不同於傳統「先寫文件再做事」的合規驅動模式，我們採取：

```
傳統模式：文件 → 合規 → 實施 → 營運
本目錄模式：營運穩定 → 防禦到位 → 體驗優化 → 自動化 → 文件驗證
```

**目標**：讓IT團隊專注於實際價值交付，而非被合規流程綁架。合規是結果，不是起點。

---

## 🎯 核心原則

### 1. 營運優先，合規驗收
- ISO 27001 是「最終驗收標準」，不是「設計起點」
- 先把系統做穩定、做安全、做好用，文件自然產生

### 2. 分層建設，逐級提升
- **L1**: 可靠性與容灾（DR/HA）- 基礎層
- **L2**: 防禦與監控（Defender/Logs）- 保護層
- **L3**: 使用者體驗（工具/流程）- 便利層
- **L4**: 工具與自動化（效率提升）- 效率層
- **L5**: 文件與合規證書（驗收層）- 驗證層

### 3. 成熟度驅動，持續改進
- 每個層級有清晰的成熟度指標（Level 1-5）
- 月度追蹤，確保穩定向上提升
- 避免「看起來合規，實際上脆弱」的假象

## 🚀 使用指南

### 第一步：理解框架
閱讀 [L1-L5_營運優先層級框架.md](L1-L5_營運優先層級框架.md)，了解：
- 5個層級的定義與優先順序
- 每個層級的成熟度指標（1-5級）
- 層級間的依賴關係
- 常見誤區

### 第二步：評估現狀
使用「現狀評估報告」模板，對標現有IT能力：
- 評估各層級目前的成熟度等級
- 識別關鍵差距與風險
- 確定優先改進項

### 第三步：制定計劃
根據評估結果，制定分階段實施計劃：
- Phase 1: L1 可靠性 + L2 監控基礎（3-6個月）
- Phase 2: L2 防禦完善 + L3 體驗優化（6-9個月）
- Phase 3: L4 自動化推進（9-12個月）
- Phase 4: L5 文件整理與認證（12-18個月）

### 第四步：執行與追蹤
使用「月度進度追蹤」儀表板：
- 每月更新各層級成熟度
- 監控關鍵指標（可用性、MTTR、滿意度等）
- 調整資源與優先級

### 第五步：持續改進
定期檢討與優化：
- 季度回顧：評估進度與成果
- 半年度審查：調整策略與資源
- 年度總結：對標業界最佳實踐

## 🔗 與ISO 27001的關係

本目錄**不是**要取代ISO 27001，而是**重新定位**ISO 27001的角色：

| 傳統認知 | 營運優先模式 |
|---------|------------|
| ISO 27001是「必須遵守的規範」 | ISO 27001是「驗證營運品質的標準」 |
| 先寫政策、再實施 | 先建系統、政策自然形成 |
| 文件是目的 | 文件是結果 |
| 合規驅動技術決策 | 營運需求驅動技術，合規驗收結果 |

**實際對應**：
- L1-L4 做好了 → 自動產生 L5 需要的證據與記錄
- ISO 27001 控制項 → 映射到各層級的實施要點
- 內部稽核 → 驗證 L1-L4 是否真正有效

詳細對應關係請參考：[與ISO27001對應關係.md](與ISO27001對應關係.md)（待建立）

## 📊 成功指標

本目錄的成功，用以下指標衡量：

### 技術指標
- ✅ L1: 關鍵系統可用性 ≥ 99.5%
- ✅ L2: 事件平均處理時間 (MTTR) ≤ 4小時
- ✅ L3: 使用者滿意度 ≥ 4/5
- ✅ L4: 日常運維自動化率 ≥ 70%
- ✅ L5: ISO 27001 認證通過

### 文化指標
- 🎯 IT團隊專注於價值交付，而非填表格
- 🎯 合規流程融入日常工作，而非額外負擔
- 🎯 文件與系統同步，而非脫節
- 🎯 技術決策基於營運需求，而非僅為合規

## ⚠️ 重要提醒

### 不要做的事
❌ **不要**先寫完所有文件再實施系統  
❌ **不要**為了ISO認證而犧牲系統穩定性  
❌ **不要**讓合規流程阻礙技術創新  
❌ **不要**跳過L1-L3直接做L5（文件）  
❌ **不要**一次性追求所有層級達到滿分

### 應該做的事
✅ **先穩定L1（不中斷、不爆炸）**  
✅ **建立L2（出事有訊號、能處理）**  
✅ **優化L3（人用起來順）**  
✅ **自動化L4（減少浪費時間）**  
✅ **最後整理L5（文件與證書自然產生）**
